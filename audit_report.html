Content Security Policy — unsafe-inline Present
security · dashboard.py CSS_BASE / JS_BODY injection

Your CSP includes 'unsafe-inline' in both script-src and style-src. This weakens XSS protection. Because dashboard.py injects CSS and JavaScript as inline strings inside the HTML, a nonce-based approach would fix this properly.

Recommended Fix

In dashboard.py, generate a random nonce per build and inject it into both the CSP meta tag and every <script> / <style> block. Replace 'unsafe-inline' with 'nonce-{nonce}'. This is a one-time refactor of the HTML generation in dashboard.py.

Google Fonts — External Dependency & GDPR Note
Geist · DM Mono · Instrument Serif loaded from fonts.googleapis.com

Fonts are loaded from Google's CDN. For an internal tool used primarily within Oman this is low-risk, but Google Fonts requests log IP addresses. If the dashboard is ever accessed by EU users, this could create GDPR obligations.

Optional Fix

Download the WOFF2 files and self-host them in docs/fonts/. Update the CSP font-src to 'self' only. This also improves load speed by ~200ms (eliminates DNS lookup + cross-origin handshake).

HIGH
GitHub PAT Stored Unencrypted in localStorage
dashboard actions: Edit, Approve, Delete, Recreate, Request Changes

When users tick "Keep token on this device", the GitHub Personal Access Token is written to localStorage in plain text. Any XSS vector (even a future one) or browser extension with broad permissions can read it. The token has repo or workflow scope — meaning an attacker could push code or trigger Actions on your behalf.

  • The dashboard acknowledges this: "stored unencrypted — only on private devices"
  • The CSP's 'unsafe-inline' in script-src makes XSS theoretically easier
Recommended Fix

Short-term: Encrypt the token before writing to localStorage using the Web Crypto API (AES-GCM with a key derived from a user-set PIN). Better: Use a fine-grained PAT scoped only to the specific repo's contents and workflows — limits blast radius if leaked. Best: Move approval/edit actions to a lightweight backend (Cloudflare Worker or GitHub App) so the dashboard never touches tokens at all.

Full Functionality Requires JavaScript — No Progressive Enhancement
Filter, Approve, Edit, Delete, Copy — all JS-gated

The dashboard correctly displays a warning when JS is disabled. For an internal operator tool this is acceptable, but the post list itself renders fine without JS — only actions are lost. The noscript message could be more prominent (currently just a small banner).

LOW
OG Image as Inline base64 Data URI
og:image · twitter:image

Both og:image and twitter:image use data:image/svg+xml,... URIs. Twitter/X's card crawler and some LinkedIn previews reject data: URIs — the image may silently fail to appear when sharing the dashboard link.

Fix

In dashboard.py, write the SVG to docs/og.svg (or convert to PNG) and reference it as https://abuali85.github.io/LINKEDIN_POSTS/og.png. Takes ~5 lines of Python change.
